# Environment Setup Gate

This skill is the canonical **Stage 3** of every phased plan. It ensures all external services are provisioned and every `.env.local` file is fully populated before feature stages or CI gates begin. Running feature work with missing or placeholder keys causes cascading failures that are hard to diagnose; this gate catches them early.

## When to Use

- Running Stage 3 of a phased plan generated by `prd-to-phased-plans`
- User runs `/sp-environment-setup-gate`
- User says "env setup", "set up environment", "populate env vars", or "provision services"
- After CI/CD scaffold is complete (`sp-ci-cd-scaffold`) and before the first feature stage

## Inputs and Preconditions

- Repository exists with at least one `apps/` or `packages/` directory.
- `sp-ci-cd-scaffold` (Stage 2) has completed: `.github/workflows/` exists.
- Clean git working tree on `main`, OR explicit user OK to proceed dirty.
- The skill has read access to all `.env.example` files in the repo.

If no `.env.example` files are found at all, stop and tell the user — this likely means the apps were not scaffolded yet. Recommend completing Stage 2 first.

## Workflow

### Phase 0 — Scan

1. Recursively find all `.env.example` files across `apps/`, `packages/`, and the repo root.
2. For each file found, read every key declared in it (ignore comment lines starting with `#`).
3. Record: file path, all declared keys, and the set of `.env.local` files that should correspond.

### Phase 1 — Service Detection

Cross-reference every discovered key against [references/known-services-catalog.md](references/known-services-catalog.md) to map keys to services.

Group keys by service. Keys that do not match any known prefix are placed in an "Other / Unknown" group and still included in the checklist.

### Phase 2 — GitHub Secrets Detection

1. Scan all `.github/workflows/*.yml` files for references to `${{ secrets.* }}`.
2. For each secret referenced, check whether it matches a key found in Phase 0.
3. Build a separate "GitHub Secrets" list: keys that CI workflows need and that the user must add via GitHub repository settings (Settings → Secrets and variables → Actions).

**Important:** this skill cannot verify GitHub secrets (no read access to org secrets). It records the user's verbal confirmation only.

### Phase 3 — Checklist Generation

Using [references/env-checklist-template.md](references/env-checklist-template.md) as the structural pattern, generate a full manual setup checklist:

- One section per detected service (ordered: Supabase → Stripe → Resend → Clerk → Auth0 → NextAuth → PostHog → Sentry → Vercel → GitHub → OpenAI → Anthropic → Other)
- Each section lists exactly which keys belong to which `.env.local` file
- A final "GitHub Secrets" section (if any secrets were detected in Phase 2)

Display the checklist to the user in full. Include direct links to provisioning consoles from the catalog.

### Phase 4 — Wait for Confirmation

After displaying the checklist, ask the user to confirm when they have completed all steps.

**HITL bubble-up rule:** if the user signals they cannot access a required service (no account, blocked by billing, etc.), set `needs_human: true` with `hitl_category: external_credentials` and surface the specific service and key as `hitl_question`. Do NOT call `ask_user_input_v0` from within a subagent context — bubble up and let the orchestrator prompt.

When the user confirms "done", proceed to Phase 5.

### Phase 5 — Verification

Dispatch the [`env-verifier`](agents/env-verifier.md) subagent (model: `haiku`, effort: low).

The verifier:
- Confirms a `.env.local` exists alongside every `.env.example`
- Confirms every key from `.env.example` is present in `.env.local` with a non-empty, non-placeholder value
- Returns a structured result (see agent spec)

**If verification passes (`status: pass`):** proceed to Phase 6 — stage complete.

**If verification fails (`status: fail`):** display a targeted report:

```
Missing .env.local files:
  - apps/web/.env.local   (needed by apps/web/.env.example)

Missing keys (apps/web/.env.local):
  - STRIPE_SECRET_KEY
  - STRIPE_WEBHOOK_SECRET

Placeholder values (apps/api/.env.local):
  - RESEND_API_KEY  (currently set to "your_key_here")
```

Then return to Phase 4 and wait for the user to fix and re-confirm. Loop until `status: pass`.

### Phase 6 — Closeout

1. Record confirmation that all `.env.local` files pass verification.
2. Record user's verbal confirmation that GitHub secrets are set (if any were listed in Phase 2).
3. Mark the stage complete in `docs/plans/00_master_checklist.md` (update the Stage 3 row to `Completed`).
4. Walk the [Completion Checklist](#completion-checklist) below and confirm every box is `[x]`.
5. Report to the user: which services were set up, which files were verified, and the total key count.

## Completion Checklist

Walk this checklist at the end of every run. The stage is not "done" until all items are checked.

[ ] All `.env.example` files in the repo have been scanned and all keys catalogued
[ ] All detected services are represented in the manual checklist displayed to the user
[ ] GitHub Secrets section was generated (or confirmed empty if no CI secrets needed)
[ ] User confirmed all provisioning steps are complete
[ ] `env-verifier` subagent returned `status: pass` with zero `missing_files`, zero `missing_keys_per_file`, and zero `placeholder_keys`
[ ] User confirmed GitHub repository secrets are set (or confirmed none are required)
[ ] Stage 3 row in `docs/plans/00_master_checklist.md` updated to `Completed`

## Hard Constraints

- **Never read, log, or display actual secret values.** The `env-verifier` only checks existence and placeholder patterns — it never surfaces real values. This skill must not work around that constraint.
- **Never write `.env.local` files.** Population is the user's manual action. This skill generates checklists and verifies — it does not provision.
- **Never skip the verification loop.** Even if the user says "I've done everything", the `env-verifier` subagent must run before the stage is marked complete.
- **Stage 3 contract.** When invoked as Stage 3 of a phased plan, this skill is the entire stage. The plan file at `docs/plans/stage_3_env_setup_gate.md` delegates to this skill. After completion, mark the stage `Completed` in `docs/plans/00_master_checklist.md`.

## Sub-agent Return Contract

When this skill is invoked as a sub-agent by the orchestrator, return:

```yaml
status: complete | failed | needs_human
summary: <one paragraph describing what services were detected, which files were verified, and the final verification outcome>

needs_human: false | true
hitl_category: null | "external_credentials"
hitl_question: null | "<which service the user cannot access and why>"

```
